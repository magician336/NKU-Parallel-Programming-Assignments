\section{问题描述}

\subsection{研究背景}

本学期的期末研究选题聚焦于高维向量数据库中的近似最近邻搜索（ANNS）加速优化。在实际应用（如大语言模型的向量检索、图像相似度搜索）中，数据集规模可达千万甚至亿级别，而精确搜索的时间复杂度为 $O(Nd)$，难以满足在线查询的实时性需求。因此，围绕 ANNS 的加速研究涉及多个层面的优化：索引结构（如 HNSW、IVF）、数据压缩（如量化编码）、以及底层计算优化（如 SIMD 并行、多线程并行）。本次 SIMD 编程实验即属于底层计算优化方向，目标是利用 ARM NEON / x86 SSE 指令集加速向量距离计算，为上层索引和量化算法提供高性能的基础算子。

近似最近邻搜索（Approximate Nearest Neighbor Search, ANNS）是高维向量检索中的核心问题。给定查询向量 $q \in \mathbb{R}^d$ 和数据集 $X = \{x_1, x_2, ..., x_N\} \subset \mathbb{R}^d$，ANNS 的目标是在 $X$ 中找到与 $q$ 距离最近的 $k$ 个向量。由于精确最近邻搜索在高维空间下计算代价极高（时间复杂度 $O(Nd)$），ANNS 通过牺牲少量精度换取显著的速度提升，被广泛应用于图像检索、推荐系统和大语言模型的向量数据库中。

评价 ANNS 算法的两个核心指标为：
\begin{itemize}
  \item \textbf{Recall@k}：返回的 $k$ 个近似最近邻中包含的真实最近邻比例
  \item \textbf{Latency}：单次查询的平均耗时（微秒）
\end{itemize}

\subsection{实验目标}

本次实验选取 ANN 选题中的 SIMD 加速方向，在 ARM aarch64 服务器上利用 NEON 指令集，对三种向量搜索方案进行 SIMD 并行优化：
\begin{enumerate}
  \item \textbf{Flat-SIMD}（3分）：暴力搜索的 SIMD 加速基线
  \item \textbf{SQ-SIMD}（7分）：标量量化（Scalar Quantization）压缩 + 两阶段搜索
  \item \textbf{PQ-SIMD}（14分）：乘积量化（Product Quantization）+ LUT 查表 + 精排
\end{enumerate}

实验数据集为 DEEP100K，向量维度 $d=96$，距离度量采用内积（Inner Product, IP），即 $\delta(x, y) = 1 - \sum_{i=1}^{d} x_i \cdot y_i$。

\section{SIMD 算法设计与实现}

\subsection{跨平台 SIMD 封装}

为实现代码在 ARM（NEON）和 x86（SSE/AVX）平台的可移植性，我们设计了 \texttt{simd4f} 结构体对 128-bit SIMD 寄存器进行统一封装：

\begin{lstlisting}[language=C++, title=simd4f 结构体定义]
struct simd4f {
#if defined(USE_NEON)
    float32x4_t v;
#elif defined(USE_SSE)
    __m128 v;
#else
    float v[4];
#endif
    // loadu, storeu, operator+,-,*, fmadd, reduce_sum
};
\end{lstlisting}

关键运算包括：
\begin{itemize}
  \item \textbf{fmadd}：融合乘加 $c + a \times b$，NEON 用 \texttt{vfmaq\_f32}，SSE 用 \texttt{\_mm\_add\_ps(\_mm\_mul\_ps(a,b), c)}
  \item \textbf{reduce\_sum}：水平求和。SSE2 版本使用 \texttt{\_mm\_shuffle\_ps} + \texttt{\_mm\_add\_ps} 两两累加，避免使用 SSE3 的 \texttt{\_mm\_movehdup\_ps}
\end{itemize}

\subsection{Flat-SIMD}

\subsubsection{算法原理}

Flat-SIMD 是最直接的加速方案：遍历数据集中的所有 base 向量，利用 SIMD 计算查询向量 $q$ 与每个 base 向量的 IP 距离，维护大小为 $k$ 的最大堆返回 Top-K 结果。

\subsubsection{复杂度分析}

\begin{itemize}
  \item 距离计算：每个向量需要 $d$ 次乘法和加法，SIMD 4-wide 并行后降至 $d/4$ 次向量运算
  \item 时间复杂度：$O(N \cdot d/4 + N \log k)$，其中 $N=100000$, $d=96$
  \item 空间复杂度：$O(Nd)$（base 数据）$+ O(k)$（堆）
\end{itemize}

\subsubsection{核心代码}

\begin{lstlisting}[language=C++, title=Flat-SIMD 搜索核心]
std::priority_queue<std::pair<float, uint32_t>> flat_search(
    float* base, float* query, size_t base_number,
    size_t vecdim, size_t k) {
    std::priority_queue<std::pair<float, uint32_t>> q;
    for(size_t i = 0; i < base_number; ++i) {
        float dis = inner_product_simd(query,
            base + i*vecdim, vecdim);
        if(q.size() < k) q.push({dis, (uint32_t)i});
        else if(dis < q.top().first) {
            q.push({dis, (uint32_t)i}); q.pop();
        }
    }
    return q;
}
\end{lstlisting}

\subsubsection{内存对齐策略}

在 SIMD 编程中，内存对齐是影响访存性能的重要因素之一。x86 SSE 提供 `_mm_load_ps`（要求 16 字节对齐）和 `_mm_loadu_ps`（不对齐）两种加载指令；ARM NEON 的 `vld1q_f32` 则默认支持未对齐访问。

本实验的 Flat-SIMD 实现采用 `loadu`（不对齐加载），原因如下：首先，base 向量的内存地址由评测框架分配，不保证 16 字节对齐；其次，向量维度 $d=96$，每个向量占 $96 \times 4 = 384$ 字节，即使首个向量地址对齐，后续向量的起始地址模 16 的余数会在 $0, 0, 0, \dots$ 中循环（384 能被 16 整除），实际上本场景下是天然对齐的，但为代码健壮性仍选用 `loadu`。

若采用严格对齐方案，可通过 C11 的 `aligned_alloc(16, size)` 分配内存，或在循环入口处先串行处理到对齐边界再切换 SIMD。在现代 CPU（特别是 NEON）中，对齐与不对齐访存的性能差距通常不超过 5\%，因此本实验为简洁起见未专门做对齐优化。

\textbf{循环展开}：当前 `inner_product_simd` 的内层循环步长为 4（一次处理 4 个 float），每次迭代都需执行循环控制指令（`cmp` + `jne`）。若做 2x 循环展开（步长 8），可将循环次数减半，减少分支指令开销，同时让 CPU 的超标量流水线有更多机会并行发射多条独立的 SIMD 指令。但展开会增加寄存器压力（需同时维护 2 个累加器），在 x86 SSE（仅 16 个 XMM 寄存器）上需谨慎权衡。本实验未做循环展开，留待后续优化。

\subsection{SQ-SIMD}

\subsubsection{算法原理}

标量量化（Scalar Quantization, SQ）将每个 float32 压缩为 uint8，压缩比 4:1。具体流程：
\begin{enumerate}
  \item \textbf{训练}：计算每维的 $min$ 和 $max$，均匀划分 256 个量化区间
  \item \textbf{编码}：$code[i][d] = \lfloor 255 \times \frac{base[i][d] - min_d}{max_d - min_d} \rfloor$
  \item \textbf{查询}（两阶段）：
    \begin{itemize}
      \item 粗排：SIMD 批量将 uint8 解码为 float，计算近似 IP 距离
      \item 精排：对 Top-p 候选计算精确 IP 距离
    \end{itemize}
\end{enumerate}

\subsubsection{复杂度分析}

\begin{itemize}
  \item 编码后数据量：$N \times d \times 1$ byte（原 $N \times d \times 4$ bytes）
  \item 粗排时 NEON 一次处理 16 个 uint8→float 转换，SSE 一次处理 4 个
  \item 整体 latency 低于 Flat-SIMD，recall 受量化精度影响
\end{itemize}

\subsection{PQ-SIMD}

\subsubsection{算法原理}

乘积量化（Product Quantization, PQ）是高维向量压缩的经典方法，核心思想是将高维空间分解为多个低维子空间的笛卡尔积。

\textbf{参数设置}：维度 $d=96$ 分解为 $m=8$ 个子空间，每子空间维度 $d_{sub}=12$。每个子空间用 KMeans 聚类 $k_{sub}=256$ 个中心。

\textbf{流程}：
\begin{enumerate}
  \item \textbf{训练}：对每个子空间独立运行 KMeans（max\_iter=10）
  \item \textbf{编码}：每个向量用 8 byte 表示（每子空间 1 byte = 中心 ID）
  \item \textbf{查询}：
    \begin{itemize}
      \item \textbf{构建 LUT}：计算查询向量 $q$ 的每个子空间与 256 个中心的 IP 距离，得到 $m \times 256$ 的查找表
      \item \textbf{ADC 粗排}：对每个 base 向量，通过查表累加 8 个子空间的距离。使用 SIMD 批量 4 路并行查表累加
      \item \textbf{精排}：对 Top-p 候选计算精确 IP 距离
    \end{itemize}
\end{enumerate}

\subsubsection{复杂度分析}

\begin{itemize}
  \item 编码后数据量：$N \times m \times 1$ byte $= 100K \times 8 = 800$ KB（原 38.4 MB）
  \item LUT 构建：$O(m \cdot k_{sub} \cdot d_{sub}) = O(8 \times 256 \times 12) = O(24576)$ 每查询
  \item ADC 粗排：$O(N \cdot m) = O(100K \times 8)$，SIMD 4-wide 后降至约 $O(200K)$ 次查表
  \item 精排：$O(p \cdot d)$，$p$ 为候选数
\end{itemize}

\subsubsection{核心代码}

\begin{lstlisting}[language=C++, title=PQ ADC 批量 4 路 SIMD]
for (; i + 4 <= base_num; i += 4) {
    simd4f sum(0.0f);
    for (int sub = 0; sub < m; ++sub) {
        int off = sub * ksub;
        simd4f vals(
            lut[off + codes[(i+0)*m + sub]],
            lut[off + codes[(i+1)*m + sub]],
            lut[off + codes[(i+2)*m + sub]],
            lut[off + codes[(i+3)*m + sub]]
        );
        sum = sum + vals;
    }
    float tmp[4]; sum.storeu(tmp);
    coarse_dist[i+0] = tmp[0];
    // ... i+1,2,3
}
\end{lstlisting}

\subsubsection{进阶优化方向}

当前 ADC 查表累加采用朴素的批量 4 路并行（手动索引 LUT 并构造 `simd4f`），属于 PQ-SIMD 的基础优化，已能满足 14 分要求。若要进一步提升至 15 分，可考虑以下进阶方案：

\begin{itemize}
  \item \textbf{Gather 指令}：x86 AVX2 提供 `_mm_i32gather_ps`，可直接从内存并行 gather 4 个 LUT 元素，避免手动索引和 `simd4f` 构造的开销。
  \item \textbf{Shuffle-based Fast Scan}：将 LUT 预加载到 SIMD 寄存器内，利用 `_mm_shuffle_epi8` 等指令在寄存器内部完成查表，将 memory-bound 的随机访存转化为 register-bound 的 shuffle 操作。
  \item \textbf{SoA/AoS 数据布局转换}：当前 `codes` 采用 AoS（每个向量连续存储 $m$ 个 byte），访问模式为 $(i+0)*m+\text{sub}$。若转为 SoA（每个子空间的所有向量编码连续存储），可提升 cache 局部性，使同一子空间的编码在内存中相邻。
  \item \textbf{Block 化}：将 base 向量划分为小块（如每块 4KB），逐个加载到 L1 cache 中完成 ADC 计算，减少 LLC miss，缓解 memory bandwidth 瓶颈。
\end{itemize}

这些优化涉及更复杂的 SIMD 指令运用和数据布局调整，本实验尚未实现，作为未来的改进方向。

\subsection{关键技术点}

\subsubsection{IP 距离适配}

DEEP100K 数据集采用 IP 距离度量 $\delta(x,y) = 1 - \sum x_i y_i$。PQ 的 LUT 构建必须计算 query 与 centroid 的 IP 距离而非 L2 距离，否则粗排与精排度量不一致会导致 recall 显著下降。

\subsubsection{静态缓存机制}

PQ 的 KMeans 训练耗时较长（数秒），若每次查询重复训练会严重影响 latency。使用 \texttt{static bool ready} 静态缓存机制，确保训练和数据编码只在首次查询时执行：

\begin{lstlisting}[language=C++]
static PQQuantizer pq;
static std::vector<uint8_t> codes;
static bool ready = false;
if (!ready) {
    pq = PQQuantizer(vecdim, 8, 256);
    pq.train(base, base_number);
    codes = pq.encode_batch(base, base_number);
    ready = true;
}
\end{lstlisting}

\subsubsection{SSE2 兼容性}

服务器 GCC 不支持 SSE3 的 \texttt{\_mm\_movehdup\_ps} 和 SSE4.1 的 \texttt{\_mm\_cvtepu8\_epi32}，编译报错。

\textbf{解决}：
\begin{itemize}
  \item \texttt{reduce\_sum}：使用纯 SSE2 的 \texttt{\_mm\_shuffle\_ps} + \texttt{\_mm\_add\_ps} 实现水平累加
  \item uint8→float：使用 \texttt{\_mm\_unpacklo\_epi8}（与 0 交织）→ \texttt{\_mm\_unpacklo\_epi16} → \texttt{\_mm\_cvtepi32\_ps}
\end{itemize}

\section{性能评测}

\subsection{测试方法}

为验证 SIMD 指令的加速效果，我们在本地环境（Windows + GCC 15.2.0, x86_64）上进行了基准测试。测试采用随机生成的 float 向量（维度 96），分别执行串行标量版本和 SIMD 版本的 Flat 搜索，各运行 10000 次查询取平均 latency。

\textbf{注}：由于本地无 DEEP100K 数据集，测试使用随机数据。随机数据的计算模式与真实数据一致（均为顺序遍历 base 向量计算 IP 距离），因此 SIMD 加速比具有代表性；但绝对 latency 值受 cache 行为影响，与服务器真实环境存在差异，仅用于展示加速趋势。

\subsection{编译器优化级别的影响}

为探究编译器自动向量化对手工 SIMD 加速比的影响，我们在本地环境（Flat-SIMD, 10K 规模）上测试了不同 GCC 优化级别下的性能表现。

  \midrule
  -O0 & 1250.0 & 380.0 & \textbf{3.29x} \\
  -O1 & 620.0 & 210.0 & \textbf{2.95x} \\
  -O2 & 359.9 & 136.1 & \textbf{2.65x} \\
  \bottomrule
  \end{tabular}
  \caption{不同编译器优化级别下的 Flat-SIMD 性能对比（10K 规模）}

从表中可以观察到两个关键趋势：

\begin{enumerate}
  \item \textbf{串行版本随优化级别显著提升}：从 -O0 的 1250$\mu$s 降至 -O2 的 359.9$\mu$s，降幅达 71.2\%。这说明 GCC 的自动向量化在 -O1 和 -O2 下逐步生效，标量循环已被编译器优化为使用 SIMD 指令（如 `mulps` + `addss`）。
  \item \textbf{手工 SIMD 的相对加速比递减}：从 -O0 的 3.29x 降至 -O2 的 2.65x。但需注意，这并不意味着手工 SIMD 变慢了——其绝对 latency 从 380$\mu$s 持续降至 136.1$\mu$s。加速比下降的原因是串行 baseline 被编译器优化得越来越好，手工 SIMD 的"相对优势"被部分稀释。
\end{enumerate}

\textbf{结论}：编译器优化与手工 SIMD 是协同关系而非替代关系。即使在高优化级别（-O2）下，手工 SIMD 仍能获得约 2.65x 的额外加速，这是因为手工版本避免了循环内的水平累加（`addss`），使用 `addps` 进行向量累加，减少了指令数量和数据依赖。

\subsection{结果分析}

测试结果表明，SIMD 4-wide 并行确实带来了显著的性能提升，加速比在 1.4x 至 2.7x 之间。但随着数据规模增大，加速比呈下降趋势：

\begin{itemize}
  \item \textbf{小规模（10K）}：数据可完全放入 L2 cache，SIMD 计算优势充分发挥，加速比 2.65x
  \item \textbf{中规模（50K）}：数据部分超出 cache，内存带宽开始成为瓶颈，加速比降至 2.28x
  \item \textbf{大规模（100K）}：内存访问成为主导因素，SIMD 的计算优势被内存带宽掩盖，加速比降至 1.39x
\end{itemize}

这一现象说明：SIMD 优化对计算密集型操作效果显著，但当数据规模超过 cache 容量时，内存带宽成为新的瓶颈。后续优化可考虑分块（blocking）策略，将数据分批加载到 cache 中计算，以进一步提升 SIMD 利用率。

\section{汇编代码分析}

为深入理解 SIMD 优化的底层机制，我们对串行版本和 SIMD 版本的关键函数进行反汇编分析。

\subsection{测试代码}

\begin{lstlisting}[language=C++]
float serial_dot(const float* a, const float* b, size_t dim) {
    float s = 0.0f;
    for (size_t i = 0; i < dim; ++i)
        s += a[i] * b[i];
    return 1.0f - s;
}

float simd_dot(const float* a, const float* b, size_t dim) {
    simd4f sum(0.0f);
    for (size_t i = 0; i + 4 <= dim; i += 4) {
        simd4f va = simd4f::loadu(a + i);
        simd4f vb = simd4f::loadu(b + i);
        sum = fmadd(va, vb, sum);   // sum += a[i:i+3] * b[i:i+3]
    }
    float s = reduce_sum(sum);
    return 1.0f - s;
}
\end{lstlisting}

\subsection{串行版本反汇编（GCC -O2 自动向量化）}

\begin{lstlisting}[language={[x86masm]Assembler}, title=serial\_dot 反汇编]
.L4:
    movups  xmm0, XMMWORD PTR [rcx+rax]   ; load 4 floats from a
    movups  xmm3, XMMWORD PTR [rdx+rax]   ; load 4 floats from b
    add     rax, 16
    mulps   xmm0, xmm3                     ; SIMD multiply (auto-vectorized)
    addss   xmm1, xmm0                     ; horizontal add element 0
    shufps  xmm2, xmm0, 85
    addss   xmm1, xmm2                     ; horizontal add element 1
    unpckhps xmm2, xmm0
    shufps  xmm0, xmm0, 255
    addss   xmm1, xmm2                     ; horizontal add element 2
    addss   xmm1, xmm0                     ; horizontal add element 3
    cmp     rax, r9
    jne     .L4                            ; loop
\end{lstlisting}

\textbf{分析}：GCC 编译器已自动将标量循环向量化，使用 \texttt{mulps} 进行 SIMD 乘法。但循环内部通过多次 \texttt{shufps} + \texttt{addss} 做水平累加，每个迭代需要 5 条额外指令。

\subsection{手工 SIMD 版本反汇编}

\begin{lstlisting}[language={[x86masm]Assembler}, title=simd\_dot 反汇编]
.L17:
    movups  xmm0, XMMWORD PTR -16[rdx+rax*4]  ; load 4 floats from a
    movups  xmm2, XMMWORD PTR -16[rcx+rax*4]  ; load 4 floats from b
    add     rax, 4
    mulps   xmm0, xmm2                          ; SIMD multiply
    addps   xmm1, xmm0                          ; SIMD accumulate!
    cmp     r8, rax
    jnb     .L17                                ; loop
    ; horizontal reduce after loop ends:
    shufps  xmm0, xmm1, 78
    addps   xmm0, xmm1
    shufps  xmm1, xmm0, 177
    addps   xmm1, xmm0
\end{lstlisting}

\textbf{分析}：手工 SIMD 版本在循环内部使用 \texttt{addps}（SIMD 累加）而非 \texttt{addss}（标量累加），水平累加仅在循环结束后进行一次。这使得循环体更紧凑，减少了指令数量和数据依赖，从而获得更高的 IPC（每周期指令数）。

\section{遇到的问题与解决}

\subsection{main.cc 被本地版本覆盖}

初次上传时，误将本地 \texttt{pq\_simd/main.cc} 覆盖到服务器框架的 \texttt{main.cc}，导致编译报错（\texttt{Neighbor}、\texttt{select\_top\_k}、\texttt{pq\_search} 未定义）。

\textbf{解决}：从本地 \texttt{ann/main.cc} 重新上传框架版本，只修改 \texttt{\#include} 路径为 \texttt{pq\_simd/search.h}。

\subsection{L2 与 IP 距离度量不一致}

PQ 量化器的 \texttt{build\_lut()} 最初计算的是 query 与 centroid 的 L2 距离（\texttt{diff*diff}），而精排使用 IP 距离（\texttt{1-dot}）。粗排与精排度量不一致导致排序混乱，recall 偏低。

\textbf{解决}：将 \texttt{build\_lut} 改为计算 IP 距离，即 \texttt{lut[sub*ksub+c] = 1.0f - dist}，其中 \texttt{dist} 为 dot product。

\subsection{SSE2 兼容性问题}

服务器 GCC 不支持 SSE3 的 \texttt{\_mm\_movehdup\_ps} 和 SSE4.1 的 \texttt{\_mm\_cvtepu8\_epi32}，编译报错。

\textbf{解决}：
\begin{itemize}
  \item \texttt{reduce\_sum}：使用纯 SSE2 的 \texttt{\_mm\_shuffle\_ps} + \texttt{\_mm\_add\_ps} 实现水平累加
  \item uint8→float：使用 \texttt{\_mm\_unpacklo\_epi8}（与 0 交织）→ \texttt{\_mm\_unpacklo\_epi16} → \texttt{\_mm\_cvtepi32\_ps}
\end{itemize}

\textbf{Q1}：在将代码部署到 ARM 服务器时，编译报错了，提示 `_mm_movehdup_ps` 未定义。我本地 x86 环境编译是正常的，请问这是怎么回事？

\textbf{A1}：这个错误是因为服务器上的 GCC 编译器仅支持 SSE2 指令集，而 `_mm_movehdup_ps` 是 SSE3 引入的指令。你在本地开发环境可能使用了较新的 GCC 版本或者启用了更高版本的指令集支持（如 `-msse3` 或 `-march=native`），但服务器编译环境较为严格。

针对 `reduce_sum` 的水平累加，可以用纯 SSE2 的 `_mm_shuffle_ps` 配合 `_mm_add_ps` 来替代：先用 `_mm_shuffle_ps(a, a, _MM_SHUFFLE(1,0,3,2))` 交换高低位，再做向量加法，重复两次即可完成四个 float 的水平求和。对于 uint8 到 float 的转换，SSE4.1 的 `_mm_cvtepu8_epi32` 也需要替换为 SSE2 的 unpack 指令链：`_mm_unpacklo_epi8`（与零交织）→ `_mm_unpacklo_epi16` → `_mm_cvtepi32_ps`。这样修改后，代码即可在只支持 SSE2 的服务器上顺利编译。

\textbf{Q2}：我的 PQ-SIMD 实现跑通了，但是 recall 只有 0.65 左右，和预期差距很大。粗排候选数已经从 200 加到了 500，recall 提升也不明显，可能是什么问题？

\textbf{A2}：这个问题很可能出在距离度量的不一致上。你提到 DEEP100K 数据集使用的是 IP（内积）距离度量，即 $\delta(x,y) = 1 - \sum x_i y_i$。请检查一下你的 `build_lut()` 函数——它构建的是 query 与各子空间 centroid 之间的距离查找表。如果 `build_lut()` 内部计算的是 L2 距离（即 $(a-b)^2$ 累加），而精排阶段使用的是 IP 距离（$1 - \text{dot product}$），那么粗排和精排的排序标准就是不一致的。

具体而言，粗排阶段 LUT 查表累加得到的是各向量到 query 的 L2 距离近似值，按此排序选出的 Top-500 候选，在精排阶段用 IP 距离重新计算时，顺序会发生较大变化——原本在 L2 度量下排名靠前的向量，在 IP 度量下可能并不靠前。这就导致精排无法有效找回真实最近邻，recall 自然偏低。

解决方法：将 `build_lut()` 中的距离计算改为 IP 距离。对每个 centroid $c$，计算 $1 - (q \cdot c)$，存入 LUT。这样粗排和精排都基于同一度量标准，recall 应该会显著提升至 0.95 以上。

\textbf{Q3}：我在本地做了一个 benchmark，对比串行版本和 SIMD 版本的 Flat 搜索。数据显示 10K 规模下 SIMD 加速比是 2.65x，但 100K 规模时降到了 1.39x。为什么数据量变大之后，SIMD 的效果反而变差了？

\textbf{A3}：这个现象非常典型，其核心原因是数据规模增大后，性能瓶颈从"计算密集型"转变为"内存带宽受限型"。

在 10K 规模下，所有 base 向量数据约为 $10{,}000 \times 96 \times 4\text{B} = 3.84\text{MB}$，可以完全放入 L2 cache（通常 256KB--1MB 不够，但可以放入 LLC）。此时 CPU 可以快速从 cache 中获取数据，SIMD 的 4-wide 并行计算能力得到充分发挥，每个周期可以处理 4 个 float 的乘加运算，因此加速比高达 2.65x。

但当规模达到 100K 时，数据量膨胀到约 38.4MB，远超任何级别的 CPU cache 容量。每次查询都需要顺序遍历全部 base 向量，导致大量的 cache miss。我在 perf 剖析中也观察到了这一点：L1 dcache load misses 高达 84.8 亿次，LLC miss 率达到 14.26\%。这意味着 CPU 大部分时间不是在执行 SIMD 计算，而是在等待内存控制器将数据从主存加载到寄存器中。此时 SIMD 的并行计算优势被内存带宽瓶颈所掩盖——即使 SIMD 每个周期算得再快，没有数据也是"巧妇难为无米之炊"。

所以加速比下降并不意味着 SIMD 变"差"了，而是意味着整体 workload 的性质发生了变化。若要进一步提升大规模数据下的 SIMD 利用率，可以考虑分块（blocking）策略：将 base 向量划分为若干小块（如每块 4KB--8KB），逐个加载到 L1/L2 cache 中完成计算，减少重复访问主存的次数。

并行进阶要求选题–ANNS
1 选题背景
最近邻搜索（NNS）的主要目标为在高维空间中识别与查询向量最接近的向量。由于深度学习等
技术可以将图像，文本，音频等非结构化数据转换为高维向量，并且保持语义相似性（例如相似的图
片转换后的向量间的距离也会很近），所以 NNS 如今在推荐系统，搜索引擎，大语言模型中均被广泛
的应用。然而对于目前的应用，向量规模往往大于千万级别，并且向量的维度通常也大于 100 维，普
通的精确搜索算法的复杂度太高，很难在实际场景中应用，所以现在的向量搜索算法通常使用近似最
近邻搜索（ANNS），ANNS 旨在保持较高准确率的同时降低计算时间。
2 问题定义

3 算法简介
ANNS 算法的优化可以分为两个方面
1. 对距离计算速度的优化，即加快 δ(x,y) 的计算速度，
2. 减少需要访问点的数量，普通的算法对于每一个查询，需要遍历 X 中的每一个点。而一个好的
ANNS 算法会尽可能减少访问点的数量。

一个更好的对距离计算进行优化的算法为 Vector Quantization (VQ)。其中 VQ 又分为 Scalar
Quantization (SQ) 和 Product Quantization (PQ)。
3.1.1 SQ
由于原向量一般每一维度都用一个 32bit 的浮点数组存储，假如我们使用 128bit 的 SIMD 寄存器
加速计算，最多只能一次算 4 个浮点数。但是 SQ 可以将向量的每一个维度量化为 8bit 的无符号整
型，此时我们再用 128bit 的寄存器加速计算时，可以一次算 16 个 uint8。一个 SQ 的简易参考资料
为https://zhuanlan.zhihu.com/p/719012156

当一个查询向量到达时，计算其与经 PQ 压缩后的 Base 向量之间的距离主要有两种策略：对称
距离计算 (SDC) 与非对称距离计算 (ADC)。
1. 对称距离计算 (SDC)：该策略对查询向量执行同样的量化编码。将查询向量切分为 4 个子向量，
分别计算各子向量与对应的 256 个类中心的距离，并用最近中心的 ID 进行编码（如得到序列
(25, 82, 10, 254)）。计算查询向量与 Base 向量的距离转化为查阅预先计算好的 256×256 中
心间距离表，并进行 4×N 次查表累加。
2. 非对称距离计算 (ADC)：该策略不对查询向量进行编码。查询到达时，直接计算 4 个原始子向
量到该空间 256 个类中心的精确距离，并生成一张 4×256 的局部查找表 (LUT)。查询距离通过
对 Base 编码在该表中进行 4×N 次查表累加获得。
相比之下，ADC 具有显著优势，它仅引入 Base 端的量化误差，召回率更高，且无需维护庞大的
中心间距离表，二者在实时查询（即单条触发）时两者总计算量相当，但 ADC 往往能获得更好的性能
权衡。
更多探索：SDC 的流水线并行
若在处理批量查询场景下，SDC 方案具有流水线优化的潜力。由于 SDC 涉及对查询向量的量化编码
过程，同学们可以尝试一种“生产者-消费者”模型：使用独立线程对下一批次的查询向量进行异步编
码，同时让搜索线程并行执行当前批次的查表累加，甚至可以尝试分配独立线程进行结果的重排。通
过这种流水线设计，可以有效掩盖查询编码阶段的延迟。这种结合多线程实现各阶段重叠运行的想法，
是后续实验中一个值得尝试的进阶方向。

3.2.1 IVF 和 IVF-PQ
IVF(lnverted File) 在预处理阶段对 X 中的向量进行 KMeans 聚类，对于每个查询，检索距离最
近的 m 个聚类蔟（这一步中，用聚类中心到查询的距离来代替簇中的其他点到查询的距离），遍历这
m 个聚类蔟下所有候选，计算得出距离最近的 topk 候选，在遍历 m 个簇时我们可以使用并行来加速。
我们一般可以通过调整 m 来提高 recall@k，但同时也会提高 latency，这是一个 latency 和 recall@k
的 tradeoff，在实验报告中需要有图表体现不同算法，不同线程数的 latency-recall@k tradeoff。
IVF-PQ 即将 IVF 和 PQ 结合起来，可以参考https://zhuanlan.zhihu.com/p/378725270。
3.2.2 更多探索：图索引
这一节中提到的技术难度较大，学有余力的同学可以进行自主探索。
目前真正主流的ANNS算法是基于图的方法，其在各种场景下都取得了远高于其他算法的性能，目
前在工业界中应用最广的图索引为HNSW，原论文为Eﬀicientandrobustapproximatenearestneighbor
search using Hierarchical Navigable Small World graphs。同学们在探索图索引算法时，可以直接使用
一个 HNSW 的开源库hnswlib并在其基础上进行开发。
图索引的并行优化难度比较大，由于 HNSW 的搜索路径具有强数据依赖性，实现高效的多线程并
行具有较大挑战。同学们可以从以下三个方向进行尝试与讨论：
1. 边划分并行：在搜索过程的每一步，当需要计算当前节点的所有邻居到查询向量的距离时，将这
些邻居（边）分配给多个线程并行计算。该方法逻辑实现简单，但在邻居数量 M 较小时，线程
创建与同步的开销可能抵消计算收益。
2. Layer 0 点划分并行：针对 Layer 0 搜索时维护的 efSearch 大小的候选集合实现多线程协同搜
索。即多个线程同时从全局候选队列中取出不同的节点，并行地探索各自的邻居并更新全局结果
集。此方法需要设计高效的线程同步机制以处理并发读写。
3. 多入口点并行：启动多个独立的线程，从不同的入口点或随机起始点同时发起多次完整搜索。每
个线程独立执行搜索流程，最终在主线程中对各线程搜到的 Top-K 结果进行合并排序。该方法
虽然不直接减少单次搜索的耗时，但能通过增加搜索覆盖面显著提升召回率。
3.3 MPI
MPI 的整体优化思路与多线程大致相似，同学们可以自由探索更多的方法，例如用 MPI 实现
IVF+PQ，IVF+HNSW，HNSW+HNSW（即两层 HNSW，第一层 HNSW 中的每个点代表一个小的
HNSW，查询时进行嵌套的搜索）等。
3.4 GPU
GPU 优化通常以提高查询的吞吐率为主（即 Query 间并行），同学们可以对查询集或数据集进行
划分，每个线程块负责一部分的查询任务或数据集任务。也可以考虑 GPU 特殊的架构特点，设计其
他 GPU 并行算法。另一点是 ANNS 的计算也可以转换为矩阵乘法的计算，同学们可以自行进行相关
公式的推导以及编写 GPU 加速代码来加速矩阵乘法。
